Auth & Security: Guarding the Kingdom

"Security isn't a feature, it's the foundation. We don't just lock the doors; we check the ID of everyone who walks in."

🔐 The Multi-Layered Auth Flow

The Gilpin Arrivals Tool uses a sophisticated hybrid authentication model to ensure both ease of use and maximum security.

  1. Initial Login: Handled by Firebase Authentication (Email/Password or Google). Access is restricted to @thegilpin.co.uk domains.
  2. Token Verification: The Firebase ID Token is sent to /api/validate-auth.
  3. HMAC Issuance: Once verified, the server generates an HMAC Session Token. This token is used for all subsequent internal API calls.

🔥 Pro Tip: This "Double-Lock" mechanism ensures that even if a client-side Firebase session persists, the API access requires a fresh, server-validated HMAC token.

🛡️ HMAC Token System

Internal API requests are secured using Hash-based Message Authentication Codes (HMAC).

  • Format: timestamp.hmac_hash
  • Algorithm: SHA-256 using a secret AUTH_SECRET.
  • Expiry: Tokens automatically expire after 8 hours.
  • Security: The server uses constant-time comparison to prevent timing attacks during verification.

👥 Role-Based Access Control (RBAC)

Not all staff members see the same data. Access is governed by both Roles and Departments.

RoleCapabilities
AdminFull system access, user management, role assignment.
StaffStandard dashboard access based on department.
ViewerRead-only access, primarily for analytics.

Departments like REC (Reception), NIGHT (Night Audit), and HOD (Heads of Department) are granted FULL_ACCESS to all dashboard tabs.

🕵️ GDPR & AI Anonymization

When guest data is sent to external AI services (like Gemini), we must protect PII (Personally Identifiable Information). The anonymize.ts utility is the silent guardian here.

The Process:

  1. Client strips sensitive fields: name, car, carDetails.
  2. Names are replaced with tokens (e.g., "Guest A").
  3. AI processes the "clean" data and returns structured results.
  4. The Client deanonymizes the results using a local name-to-token map before showing them to the user.

🔥 Firebase Security Rules

The Realtime Database isn't a "wild west". Access is strictly enforced via database.rules.json.

{
  "sessions": {
    "$sessionId": {
      ".read": "auth != null && (auth.token.role === 'admin' || auth.token.role === 'staff')",
      ".write": "auth != null && auth.token.role === 'admin'"
    }
  }
}

Rules verify the user's Custom Claims (set via the Admin Panel) before allowing any read/write operation.

🚦 Live Token Safety

How does the browser get the Gemini API key without it being hardcoded? It calls /api/live-token.

This endpoint is the most protected in the app:

  • Requires a valid HMAC Bearer token.
  • Enforces Origin Validation (only allowing requests from the official app domain).
  • Uses Cache-Control: no-store to prevent the key from being cached by proxies.
