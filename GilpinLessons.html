🛠️ What Is the Gilpin Arrivals Tool?

A real-time operational dashboard for Gilpin Hotel staff. It replaces manual paper-based check-in processes by parsing the hotel's PDF arrival sheets and presenting them as a live, collaborative, interactive interface.

  • Receptionists see who's arriving, their room, dietary needs, and car registration.
  • Night staff use it for the morning handover — a snapshot of the previous night.
  • Managers can lock a session once it's finalised to prevent edits.

🛡️ Issue #2 — Security

Auth & Security:
Guarding the Kingdom

🕒 Lesson 2 of 10 🏨 Gilpin Hotel ✔️ Completed
"Security isn't an add-on. It's the foundation. Every route, every write, every read — trust nothing until it's verified." — Gilpin Arrivals Tool, Security Architecture

🔐 Firebase Authentication

The tool uses Firebase Authentication as its identity layer. Staff log in with email and password — no social logins, no magic links. Clean, simple, auditable.

  • Session persistence: Firebase keeps users logged in across browser sessions using secure tokens stored in IndexedDB.
  • Automatic token refresh: Firebase handles token expiry silently — staff never get logged out mid-shift.
  • Auth state listener: onAuthStateChanged reactively updates the entire UI when login state changes.

👥 Role-Based Access Control (RBAC)

Not all staff are equal. The tool enforces roles using Firebase Custom Claims — metadata attached to the user's JWT token at the server level.

RoleCan ViewCan EditCan Lock
Admin✅✅✅
Staff✅✅❌
Viewer❌ (Revoked)❌❌

Note: Viewer access was recently revoked at the database level. Only explicitly designated Staff and Admins can view session data now.

Custom claims are set via the Firebase Admin SDK and are verified on both the client and in Firebase Security Rules — so they cannot be spoofed.

🛡️ Firebase Security Rules (Recently Hardened)

The Realtime Database is locked down by rules that run server-side. Recent security audits introduced several strict enforcements:

  • Strict RBAC & Auth: All read and write operations explicitly require auth != null AND auth.token.email != null (blocking anonymous auth completely), AND the role must be exactly staff or admin.
  • Payload Validation: Every single field (like guest notes, room names, etc.) is strictly validated for type and length (e.g., newData.isString() && newData.val().length <= 5000) to prevent payload injection or database bloat.
  • Asymmetric Access: Nodes like heartbeat are write-only (reads are blocked to prevent scraping), while analytics.snapshots are read-only (prevents client-side tampering).
  • Default Deny: A catch-all $other rule explicitly blocks any reads or writes to undefined nodes.

🛤️ Route Guards

On the client side, React routes are wrapped in auth guard components that check the user's state before rendering:

➔

➔

If any check fails, the user is redirected to the login screen. This prevents even a quick flash of protected content.

⚠️ Common Security Pitfalls

  • Client-only role checks: Never trust the client alone. Always back it up with Security Rules.
  • Hardcoded secrets: API keys live in .env files — never committed to git. The Firebase config is safe to expose (it's public by design), but the Admin SDK key is not.
  • Stale claims: Custom claims are cached in the JWT. After updating a role, call user.getIdToken(true) to force a refresh.

📄 Issue #3 — Data Pipeline

The PDF Parser:
Turning Paper into Data

🕒 Lesson 3 of 10 🏨 Gilpin Hotel ✔️ Completed
"Data is the new oil, but raw PDFs are like unrefined crude. Our parser is the refinery that turns chaotic text into operational intelligence." — Gilpin Arrivals Tool, Architecture Notes

🏗️ The Three-Stage Pipeline

The journey from a PDF file to a live dashboard session involves three critical stages in the PDFService.

  1. File Upload: The raw PDF is received and buffered for processing.
  2. Raw Text Extraction: Using pdfjs-dist, the system extracts text while preserving positional information — the X/Y coordinates of every character.
  3. Structural Synthesis: The raw strings are mapped into a GuestData model, identifying room numbers, names, and statuses.

💡 Unlike simple text scrapers, our parser understands the visual layout of the page — not just the words on it.

📍 Coordinate-Based Parsing

In pdfService.ts, we don't just search for text — we look at where it sits on the page.

  • Column Mapping: Precise X/Y coordinates define where the "Room" column ends and the "Guest Name" column begins.
  • Horizontal Alignment: Text sharing the same Y-axis is grouped into a single guest record.
  • Resilience: Handles PDFs with varying margins far better than naive line-by-line parsing.

🔍 Regex: The Pattern Matchers

Even with coordinates, Regular Expressions validate and extract specific data types.

PurposePattern
Dates/(\d{2}\/\d{2}\/\d{4})/g
Guest EntriesMatches standard arrival formats to trigger object creation.

Regex ensures room 101 isn't misread as a date fragment or part of a guest name.

☀️ Morning Briefing vs. Arrivals

The app contains two distinct parsers for two document types:

  • Arrivals Parser: Individual guest records, car registrations, room assignments.
  • Morning Briefing Parser: High-level forecasting, restaurant diary entries, hotel statistics.

Each uses a unique set of regex rules and coordinate maps tailored to its document's specific layout.

🔄 Session Management & Sync

Once parsed, data is pushed to Firebase Realtime Database to create a collaborative session.

  1. Unique Key: Sessions are keyed by the arrival date — one session per day.
  2. Presence: /presence/{sessionId} tracks who is currently viewing.
  3. Locking: Admins set a flag in RTDB to prevent further edits after morning handover.

⚠️ The Big Three: Common Failure Modes

  • Format Inconsistency: PMS exports a PDF with slightly shifted columns, causing coordinate misses. Most common.
  • Regex Failure: Unusual guest name or note doesn't match the expected pattern.
  • Sync Lag: Large PDFs (100+ guests) hitting Firebase simultaneously cause temporary UI stutters.

🔥 Issue #4 — The Engine

Firebase RTDB:
The Real-Time Engine

🕒 Lesson 4 of 10 🏨 Gilpin Hotel ✔️ Completed
"In a fast-paced hotel environment, a delay of five seconds is an eternity. RTDB ensures that when a guest arrives at the gate, the front desk knows before they even park." — Gilpin Arrivals Tool, Performance Specs

⚡ Sync vs. Fetch: The Paradigm Shift

Traditional apps use REST APIs (Request ➔ Response). The Gilpin tool uses Data Synchronization.

  • No Polling: The client doesn't ask "Is there new data?". The server pushes data the millisecond it changes.
  • WebSockets: A persistent connection stays open between the browser and Firebase.
  • Latency: Updates typically propagate to all connected staff in <200ms.

📂 The Data Tree

Unlike SQL tables, RTDB is one giant JSON tree. We structure it for speed and security:

PathPurpose
/sessions/{date}The core guest list and statuses for a specific day.
/presence/{sessionId}Ephemeral data: who is currently looking at this session.
/bulletinsGlobal announcements and system-wide alerts.
/handoverShift-change notes and stayover calculations.

🤝 Presence Tracking: Who's Online?

The "Presence" system uses a special Firebase feature called onDisconnect.

  1. When a staff member opens the dashboard, they write their UID to /presence/{sessionId}/{uid}.
  2. The client registers an onDisconnect().remove() operation.
  3. If the browser closes, the tab crashes, or the Wi-Fi drops, Firebase itself deletes the record after a short timeout.

This allows managers to see exactly which departments are currently monitoring the arrivals.

🔒 Server-Side Enforcement

We don't trust the client to validate data. Our Security Rules act as a server-side firewall:

  • Auth Guard: auth != null ensures only logged-in staff can see anything.
  • Role Check: auth.token.role === 'admin' is required for destructive actions like deleting a session.
  • Schema Validation: Rules check that newData.child('room').isString() and length is valid, preventing malformed data from entering the tree.

📶 Offline Resilience

Hotel Wi-Fi can be spotty. Firebase handles this gracefully:

  • Local Cache: Changes are written to a local buffer immediately (UI feels instant).
  • Background Sync: Once the connection is restored, Firebase merges the local changes back to the server.
  • Conflict Resolution: The "Last Write Wins" policy ensures the most recent edit is preserved.

🌐 Issue #8 — Global Standards

Advanced Integration:
Global Standards & AI Resilience

🕒 Lesson 8 of 10 🏨 Gilpin Hotel ✨ Latest Developments
"Building for tomorrow means anticipating global demands today. Our AI integrations are not just smart; they're secure, compliant, and architected for worldwide excellence." — Gilpin Arrivals Tool, Global Strategy

🧠 Strategic AI Model Orchestration

The Gilpin Arrivals Tool employs a sophisticated strategy for integrating AI, carefully selecting models based on task complexity and cost-efficiency. This ensures optimal performance without unnecessary resource consumption.

  • Gemini 3 Pro: Utilized for high-complexity tasks such as refining guest records with detailed PMS notes and generating bespoke itineraries. Its advanced reasoning capabilities are crucial here.
  • Gemini 2.5 Flash: Powers high-volume, lower-complexity tasks including analytics generation, sentiment analysis from guest notes, cleaning order suggestions, upgrade suggestions, and collating reviews. This model provides speed and efficiency.
  • Cost-Efficiency: This tiered model approach allows for a balance between processing power and operational costs, ensuring that advanced AI is used where it truly adds value.

🔒 End-to-End Data Anonymization & GDPR Compliance

Protecting guest privacy is paramount. The tool implements a robust anonymization layer to ensure GDPR and global data protection standards are met.

  • PII Stripping: Before any sensitive guest data (names, car registrations, booking stream details) is sent to AI endpoints, it is meticulously stripped and anonymized using the anonymize.ts utility.
  • Tokenization: Real guest information is temporarily replaced with generic tokens (e.g., "Guest A") during AI processing.
  • Local Restoration: Upon receiving AI-processed results, the original guest information is safely restored locally using a secure name mapping system, ensuring PII is never exposed to external AI models.
  • Firebase Security Rules: Further reinforced by server-side Firebase Security Rules that strictly control access based on roles and department, preventing unauthorized exposure.

🛡️ Robust Session Management & API Security

Ensuring secure and seamless operations, the tool implements advanced session management and API protection mechanisms.

  • HMAC Session Tokens: Beyond Firebase ID Tokens, API requests are secured with server-generated HMAC session tokens (format: timestamp.hmac). These tokens are SHA-256 hashed with a secret key (process.env.AUTH_SECRET) and have an 8-hour expiry, protecting against token spoofing and replay attacks.
  • Firebase Authentication & RBAC: Users authenticate via Firebase, and custom claims enforce Role-Based Access Control (admin, staff, viewer) and department-specific permissions.
  • CORS & Origin Validation: Cross-Origin Resource Sharing (CORS) is strictly enforced with whitelisted origins (localhost, *.vercel.app, custom domains) at the API level, preventing unauthorized cross-origin requests.
  • Live Token Endpoint: The endpoint serving sensitive API keys (e.g., Gemini API key) is protected by HMAC Bearer tokens, origin validation, and strict no-cache headers.

⚡ Real-Time AI & Resilient Data Pipelines

The tool is designed for high availability and responsiveness, incorporating real-time AI capabilities and robust error handling for data pipelines.

  • Live Voice Assistant: Features a real-time WebSocket connection for audio streaming, enabling instantaneous transcription and department-specific AI interactions. This ensures dynamic, contextual conversations with AI.
  • Batch Processing & Retry Logic: Guests are processed in manageable chunks, and requests that encounter transient errors are retried with exponential backoff, preventing system overload and ensuring data integrity.
  • Comprehensive Error Handling: The geminiService.ts implements robust error capturing, logging, and user-friendly notifications for API failures.
  • PDF Parser Resilience: Despite format inconsistencies being the most common failure mode, coordinate-based parsing and strict regex patterns ensure the data pipeline remains resilient, extracting structured guest data reliably.

🎙️ Issue #10 — Future-Proofing

Real-Time Voice:
The WebSocket Frontier

🕒 Lesson 10 of 10 🏨 Gilpin Hotel ✨ New Feature
"Voice is the most natural interface. By bridging the gap between sound and data in real-time, we create a tool that listens as well as it leads." — Gilpin Arrivals Tool, Voice Integration

🎙️ Live Voice Assistant

The latest evolution of the Gilpin tool is the Live Voice Assistant, designed to provide a hands-free interface for busy hotel staff.

  • WebSocket Connection: Manages audio streaming for voice interaction, facilitating real-time communication with the AI.
  • Transcription Flow: Real-time audio input is transcribed into text, allowing for dual processing of voice and text inputs.
  • Department-Specific Prompt Context: Creates tailored conversations based on guest context and department needs, enhancing the accuracy of responses.

⚡ The WebSocket Advantage

Traditional HTTP requests are too slow for natural conversation. We use WebSockets to achieve the low latency required for voice:

  • Full-Duplex: Data flows in both directions simultaneously, allowing the AI to "listen" while it "thinks".
  • Streaming Audio: Audio is sent in small chunks, meaning transcription and processing start the moment the staff member begins speaking.
  • UI Synchronization: The voice assistant can trigger UI changes (like opening a specific guest's record) instantly via the WebSocket channel.

🔒 Securing the Voice Stream

Streaming audio and accessing powerful AI models requires rigorous security measures:

  • Live Token Endpoint: The /api/live-token endpoint serves the Gemini API key, protected by HMAC Bearer tokens and origin validation.
  • No-Cache Headers: Strict headers prevent caching of sensitive API keys by proxies and CDNs.
  • GDPR Compliance: Just like our text pipelines, the anonymize.ts utility strips PII from the context before it's sent to the AI for voice processing.

🛠️ Integration Patterns

Voice is built into the same robust framework as the rest of the tool:

  • Shared Client Patterns: Voice features follow the same error handling and retry logic as the geminiService.ts.
  • Exponential Backoff: If the WebSocket connection drops, the client automatically attempts to reconnect with increasing delays.
  • User Feedback: Visual indicators show exactly when the system is listening, transcribing, or speaking back to the staff.

🧠 AI Features & Model Usage

The Gilpin Arrivals Tool integrates advanced AI capabilities to enhance guest management and operational efficiency, strategically using different Gemini models:

  • Complex Tasks (Gemini 3 Pro): Used for high-reasoning tasks like Refine (enriching guest records from PMS notes) and Itinerary Generation (creating bespoke guest itineraries).
  • High-Volume Tasks (Gemini 2.5 Flash): Powers operations such as Analytics (generating operational insights), Sentiment Analysis (auto-tagging guests based on notes), Cleaning Order (suggesting optimal priorities), Upgrade Suggestions, and Reviews Fetcher (collating ratings from platforms).
  • Data Anonymization: Personally Identifiable Information (PII) is stripped and tokenized by the anonymize.ts utility before being sent to AI endpoints, ensuring GDPR compliance.

📄 Data Pipeline & Session Management

The tool's robust data pipeline transforms raw PDF documents into actionable, real-time guest data, managed through efficient session mechanisms:

  • PDF Parsing: Utilizes pdfjs-dist for raw text extraction and coordinate-based parsing, along with specific regex patterns (e.g., /(\d{2}\/\d{2}\/\d{4})/g for dates), to derive structured guest data from PDF arrival sheets.
  • Guest Data Model: Includes key fields such as id, name, room, status, and notes, facilitating comprehensive guest management.
  • Session Management: Sessions are created, joined, locked, and synchronized across devices using Firebase Realtime Database (RTDB). Presence tracking (/presence/{sessionId}) ensures real-time user visibility.

🔐 Authentication, RBAC & Security

Security is a paramount concern, implemented through multi-layered authentication and authorization mechanisms:

  • Authentication Flow: Users log in via Firebase Authentication (email/password or Google sign-in for @thegilpin.co.uk domain). Successful login generates an HMAC session token (timestamp.hmac, SHA-256 with process.env.AUTH_SECRET) for API requests, expiring after 8 hours.
  • Role-Based Access Control (RBAC): Defines roles (admin, staff, viewer) and departments (e.g., REC, NIGHT, HK). Access is enforced via Firebase Custom Claims and strict Firebase Security Rules, which validate roles, email domains, and data types/lengths for all read/write operations.
  • CORS & Origin Validation: Cross-Origin Resource Sharing is enforced, whitelisting specific origins (localhost, *.vercel.app, custom domains) to prevent unauthorized access. The /api/live-token endpoint (serving Gemini API keys) is protected by HMAC tokens and origin validation.

🖥️ UI/Dashboard Overview

The user interface is designed for dynamic, department-specific operational efficiency and real-time collaboration:

  • ViewManager: Dynamically renders 9 distinct dashboard tabs (e.g., Reception, Housekeeping, Handover Hub, Admin Panel) based on the user's department and role, providing tailored experiences.
  • GuestRow: Features inline editing fields and status flags (VIP, dietary needs) for immediate updates, which sync across all connected staff members in milliseconds.
  • Handover Hub: A specialized view for shift transitions, offering stayover detection and AI-generated operational briefings.
  • Department Access Control: Beyond server-side rules, the UI dynamically hides navigation buttons and restricts views based on user permissions, ensuring staff only see relevant data.
