The PDF Parser: Turning Paper into Data

"Data is the new oil, but raw PDFs are like unrefined crude. Our parser is the refinery that turns chaotic text into operational intelligence."

🚀 The Three-Stage Pipeline

The journey from a PDF file to a live dashboard session involves three critical stages implemented in the PDFService.

  1. File Upload: The raw PDF is received and buffered for processing.
  2. Raw Text Extraction: Using pdfjs-dist, the system extracts text while preserving positional information (coordinates).
  3. Structural Synthesis: The raw strings are mapped into a GuestData model, identifying room numbers, names, and statuses.

💡 Note: Unlike simple text scrapers, our parser understands the visual layout of the page.

📍 Coordinate-Based Parsing

In pdfService.ts, we don't just search for text; we look at where it is on the page.

  • Column Mapping: Precise X/Y coordinates define where the "Room" column ends and the "Guest Name" column begins.
  • Horizontal Alignment: Text on the same Y-axis is grouped into a single guest record.
  • Resilience: This method handles PDFs with varying margins much better than simple line-by-line parsing.

🔍 Regex: The Pattern Matchers

Even with coordinates, we need Regular Expressions to validate and extract specific data types.

Regex ensures that a room number like 101 isn't confused with a date or a partial guest name.

☕ Morning Briefing vs. Arrivals

The app actually contains two distinct parsers to handle different document types:

  • Arrivals Parser: Focuses on individual guest records, car regs, and room assignments.
  • Morning Briefing Parser: Focuses on high-level forecasting, restaurant diary entries, and overall hotel stats.

Each uses a unique set of regex rules and coordinate maps tailored to the document's specific design.

🏗️ Session Management & Sync

Once parsed, the data is pushed to Firebase Realtime Database to create a collaborative session.

The Sync Logic:

  1. Unique ID: Sessions are keyed by the arrival date.
  2. Presence: /presence/{sessionId} tracks who is currently viewing the data.
  3. Locking: Admins can "Lock" a session, which updates a flag in RTDB to prevent further edits after the morning handover.

⚠️ Common Failure Modes

Parsing isn't magic. When data goes missing, it's usually due to one of these "The Big Three" issues:

  • Format Inconsistency: The PMS exports a PDF with slightly shifted columns, causing coordinate misses.
  • Regex Failure: A guest has a name or note that doesn't match the expected pattern.
  • Sync Lag: Large PDFs (100+ guests) hitting the database simultaneously can cause temporary UI stutters.
