The Big Picture:
Architecture & Overview

📚 Lesson 1 of 5 🏨 Gilpin Hotel ✅ Completed
"Before you write a single line, understand the system. Every great tool starts with a clear picture of the whole." — Gilpin Arrivals Tool, Architecture Notes

🗺️ What Is the Gilpin Arrivals Tool?

A real-time operational dashboard for Gilpin Hotel staff. It replaces manual paper-based check-in processes by parsing the hotel's PDF arrival sheets and presenting them as a live, collaborative, interactive interface.

  • Receptionists see who's arriving, their room, dietary needs, and car registration.
  • Night staff use it for the morning handover — a snapshot of the previous night.
  • Managers can lock a session once it's finalised to prevent edits.

🛠️ The Tech Stack

Every technology was chosen deliberately — here's why each one earns its place:

Why Firebase? The real-time database allows multiple staff members to see updates instantly — no page refresh, no polling. When one receptionist checks a guest in, everyone else sees it within milliseconds.

🏨 Lake House vs Main Hotel

Gilpin is actually two properties — the Main Hotel and the Lake House. The tool handles them separately:

  • Each property has its own room numbering convention and PDF layout.
  • The parser detects which property a PDF belongs to automatically.
  • Dashboards can be filtered by property so staff don't see irrelevant data.

📰 Issue #2 · Security

Auth & Security:
Guarding the Kingdom

📚 Lesson 2 of 5 🏨 Gilpin Hotel ✅ Completed
"Security isn't an add-on. It's the foundation. Every route, every write, every read — trust nothing until it's verified." — Gilpin Arrivals Tool, Security Architecture

🔑 Firebase Authentication

The tool uses Firebase Authentication as its identity layer. Staff log in with email and password — no social logins, no magic links. Clean, simple, auditable.

  • Session persistence: Firebase keeps users logged in across browser sessions using secure tokens stored in IndexedDB.
  • Automatic token refresh: Firebase handles token expiry silently — staff never get logged out mid-shift.
  • Auth state listener: onAuthStateChanged reactively updates the entire UI when login state changes.

??? Firebase Security Rules (Recently Hardened)

The Realtime Database is locked down by rules that run server-side. Recent security audits introduced several strict enforcements:

  • Strict RBAC & Auth: All read and write operations explicitly require auth != null AND auth.token.email != null (blocking anonymous auth completely), AND the role must be exactly staff or admin.
  • Payload Validation: Every single field (like guest notes, room names, etc.) is strictly validated for type and length (e.g., newData.isString() && newData.val().length <= 5000) to prevent payload injection or database bloat.
  • Asymmetric Access: Nodes like heartbeat are write-only (reads are blocked to prevent scraping), while analytics.snapshots are read-only (prevents client-side tampering).
  • Default Deny: A catch-all $other rule explicitly blocks any reads or writes to undefined nodes.

🚪 Route Guards

On the client side, React routes are wrapped in auth guard components that check the user's state before rendering:

→

→
Role Check
→

If any check fails, the user is redirected to the login screen. This prevents even a quick flash of protected content.

⚠️ Common Security Pitfalls

  • Client-only role checks: Never trust the client alone. Always back it up with Security Rules.
  • Hardcoded secrets: API keys live in .env files — never committed to git. The Firebase config is safe to expose (it's public by design), but the Admin SDK key is not.
  • Stale claims: Custom claims are cached in the JWT. After updating a role, call user.getIdToken(true) to force a refresh.

The PDF Parser:
Turning Paper into Data

📚 Lesson 3 of 5 🏨 Gilpin Hotel ✅ Completed
"Data is the new oil, but raw PDFs are like unrefined crude. Our parser is the refinery that turns chaotic text into operational intelligence." — Gilpin Arrivals Tool, Architecture Notes

🚀 The Three-Stage Pipeline

The journey from a PDF file to a live dashboard session involves three critical stages in the PDFService.

  1. File Upload: The raw PDF is received and buffered for processing.
  2. Raw Text Extraction: Using pdfjs-dist, the system extracts text while preserving positional information — the X/Y coordinates of every character.
  3. Structural Synthesis: The raw strings are mapped into a GuestData model, identifying room numbers, names, and statuses.

💡 Unlike simple text scrapers, our parser understands the visual layout of the page — not just the words on it.

📍 Coordinate-Based Parsing

In pdfService.ts, we don't just search for text — we look at where it sits on the page.

  • Column Mapping: Precise X/Y coordinates define where the "Room" column ends and the "Guest Name" column begins.
  • Horizontal Alignment: Text sharing the same Y-axis is grouped into a single guest record.
  • Resilience: Handles PDFs with varying margins far better than naive line-by-line parsing.

🔍 Regex: The Pattern Matchers

Even with coordinates, Regular Expressions validate and extract specific data types.

PurposePattern
Dates/(\d{2}\/\d{2}\/\d{4})/g
Guest EntriesMatches standard arrival formats to trigger object creation.

Regex ensures room 101 isn't misread as a date fragment or part of a guest name.

☕ Morning Briefing vs. Arrivals

The app contains two distinct parsers for two document types:

  • Arrivals Parser: Individual guest records, car registrations, room assignments.
  • Morning Briefing Parser: High-level forecasting, restaurant diary entries, hotel statistics.

Each uses a unique set of regex rules and coordinate maps tailored to its document's specific layout.

🏗️ Session Management & Sync

Once parsed, data is pushed to Firebase Realtime Database to create a collaborative session.

  1. Unique Key: Sessions are keyed by the arrival date — one session per day.
  2. Presence: /presence/{sessionId} tracks who is currently viewing.
  3. Locking: Admins set a flag in RTDB to prevent further edits after morning handover.

⚠️ The Big Three: Common Failure Modes

  • Format Inconsistency: PMS exports a PDF with slightly shifted columns, causing coordinate misses. Most common.
  • Regex Failure: Unusual guest name or note doesn't match the expected pattern.
  • Sync Lag: Large PDFs (100+ guests) hitting Firebase simultaneously cause temporary UI stutters.

📰 Issue #4 · The Engine

Firebase RTDB:
The Real-Time Engine

📚 Lesson 4 of 5 🏨 Gilpin Hotel 🔴 Current
"In a fast-paced hotel environment, a delay of five seconds is an eternity. RTDB ensures that when a guest arrives at the gate, the front desk knows before they even park." — Gilpin Arrivals Tool, Performance Specs

📡 Sync vs. Fetch: The Paradigm Shift

Traditional apps use REST APIs (Request → Response). The Gilpin tool uses Data Synchronization.

  • No Polling: The client doesn't ask "Is there new data?". The server pushes data the millisecond it changes.
  • WebSockets: A persistent connection stays open between the browser and Firebase.
  • Latency: Updates typically propagate to all connected staff in <200ms.

🌳 The Data Tree

Unlike SQL tables, RTDB is one giant JSON tree. We structure it for speed and security:

PathPurpose
/sessions/{date}The core guest list and statuses for a specific day.
/presence/{sessionId}Ephemeral data: who is currently looking at this session.
/bulletinsGlobal announcements and system-wide alerts.
/handoverShift-change notes and stayover calculations.

👥 Presence Tracking: Who's Online?

The "Presence" system uses a special Firebase feature called onDisconnect.

  1. When a staff member opens the dashboard, they write their UID to /presence/{sessionId}/{uid}.
  2. The client registers an onDisconnect().remove() operation.
  3. If the browser closes, the tab crashes, or the Wi-Fi drops, Firebase itself deletes the record after a short timeout.

This allows managers to see exactly which departments are currently monitoring the arrivals.

🔐 Server-Side Enforcement

We don't trust the client to validate data. Our Security Rules act as a server-side firewall:

  • Auth Guard: auth != null ensures only logged-in staff can see anything.
  • Role Check: auth.token.role === 'admin' is required for destructive actions like deleting a session.
  • Schema Validation: Rules check that newData.child('room').isString() and length is valid, preventing malformed data from entering the tree.

📶 Offline Resilience

Hotel Wi-Fi can be spotty. Firebase handles this gracefully:

  • Local Cache: Changes are written to a local buffer immediately (UI feels instant).
  • Background Sync: Once the connection is restored, Firebase merges the local changes back to the server.
  • Conflict Resolution: The "Last Write Wins" policy ensures the most recent edit is preserved.
